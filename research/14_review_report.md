## 1. Value Proposition レビュー

**総合評価: 4 / 5（良好）**

**対象ファイル**: 02_scalardb_usecases.md, 07_transaction_model.md, 13_scalardb_317_deep_dive.md

### 1.1 良い点

- **核心的価値の明確な表現**: 「データベースの異種性を抽象化し、異種DB間でACIDトランザクションを提供する」という核心がドキュメント全体で一貫して強調されている。XA非対応のNoSQLを含む環境でこそ真価を発揮するという差別化ポイントが明確
- **デシジョンツリーの構成が実用的**: Q1からQ12まで段階的に判断を分岐させる構成で、プリセールス段階で実際に使える水準。「ScalarDB不要」のケースを明示的に挙げている点は信頼性を高める誠実なアプローチ
- **XA/Saga/TCCとの比較が体系的**: 伝統的2PC（XA）、Sagaパターン（Choreography/Orchestration）、TCCパターンそれぞれについて、ScalarDBなし/ありの比較表が整備されている。特にSagaとの関係を「代替」と「強化」の2パターンに分けて説明している点は秀逸
- **3.17新機能の技術的深度**: Piggyback Begin + Write Bufferingの動作メカニズム図が分かりやすく、ベンチマーク結果（Indirect modeで最大約2倍改善）の提示も説得力がある
- **「不要ケース」の誠実な記述**: 「単一RDBMSで完結」「結果整合性で十分」「XAで対応可能」等の不要ケースを明記しており、過度なセールストークを避ける姿勢
- **コード例と設定例の充実**: 各ユースケースにJavaコード例、設定ファイル例が豊富に含まれている
- **VLDB 2023論文への参照**: アカデミックなバックグラウンドによるConsensus Commitプロトコルの学術的裏付け

### 1.2 改善が必要な点

#### 1.2.1 Consensus Commitの分離レベル記述に不整合がある

**問題箇所**: 07_transaction_model.md セクション1.3「分離レベル: Snapshot Isolation（SI）およびSerializableをサポート」

**修正提案**: ScalarDBのConsensus Commitが提供する分離レベルはSnapshot IsolationとExtra-Read（anti-dependency checkによるSerializable相当）である。「Strict Serializability」という表現（ユースケース分類マトリクス内）は、Consensus CommitのSerializableモードがStrict Serializabilityを完全に保証するかどうか慎重に確認が必要。VLDB論文の記述に沿って正確な保証レベルを明記すべき。

#### 1.2.2 ScalarDB 2PCの「マスターレス」表現が誤解を招く

**問題箇所**: 07_transaction_model.md セクション2.2、3.2「マスターレスアーキテクチャ」「クライアント協調型、マスターレス」

**修正提案**: 2PCインターフェースにはCoordinatorサービスとParticipantサービスの明確な役割分担が存在する。「専用のコーディネータプロセスが不要（コーディネーション状態をデータベース上で管理）」のように、より精緻な表現に修正すべき。

#### 1.2.3 伝統的2PC（XA）との比較表にバイアスがある

**問題箇所**: 07_transaction_model.md セクション3.2の比較表

**修正提案**: OCCはコンフリクト時にリトライが発生するため、高競合環境ではロックベースの2PCに劣る場合がある。比較表に「高競合時のリトライコスト」の行を追加し、ScalarDB側の弱点も記載すべき。「パフォーマンス」の記述は「低〜中競合環境でOCCにより高スループット」と条件付きにすべき。

#### 1.2.4 「可用性」に関する比較が不足・不公正

**問題箇所**: 07_transaction_model.md セクション3.1で伝統的2PCの限界として「全参加者が利用可能でなければトランザクションが開始できない」を挙げているが、ScalarDB 2PCも同様の制約を持つ

**修正提案**: 比較表に「可用性」の行を追加し、ScalarDB 2PCでも「全参加者が利用可能であることが必要」と明記すべき。

#### 1.2.5 Virtual Tablesの説明がドキュメント間で乖離

**問題箇所**: 02_scalardb_usecases.md「マイクロサービス間のデータ参照パターンがさらに柔軟になります」

**修正提案**: Virtual Tablesの主目的は「Transaction Metadata Decoupling」であり、「マイクロサービス間のデータ参照パターンの柔軟化」は不正確で過大評価。「既存テーブルのスキーマ変更なしでScalarDB管理下に取り込むためのストレージ抽象」と正確に記載すべき。Virtual Tablesは任意の2テーブルを自由にJOINする機能ではなく、「データテーブル + メタデータテーブル」のINNER/LEFT OUTER JOINに限定されている。

#### 1.2.6 オブジェクトストレージ対応の過大評価

**問題箇所**: 02_scalardb_usecases.md「大容量データやコールドデータの管理にScalarDBのトランザクション保証を適用できる可能性が開けます」

**修正提案**: Private Previewの機能について具体的なユースケースやスコープを示さない限り、読者に過度な期待を持たせる恐れがある。「詳細は公式ドキュメントを参照」等の注記を追加すべき。

#### 1.2.7 業種別具体例が不足

**修正提案**: 以下の業種別ユースケースシナリオを追加すべき:
- **金融**: 口座間送金、決済ゲートウェイ
- **EC/小売**: 在庫管理+受注+決済の統合トランザクション
- **医療**: 電子カルテ+処方箋+保険請求
- **ゲーム**: マルチリージョンでのアイテム取引・課金

#### 1.2.8 CDC連携時のScalarDBメタデータの影響が過小評価

**問題箇所**: 07_transaction_model.md セクション8.3「ScalarDBのメタデータを考慮したフィルタリングが必要」

**修正提案**: PREPARED状態のレコード変更を「確定済み」と誤認する深刻な問題の具体的な注意喚起が必要。3.17のTransaction Metadata Decoupling適用時のCDC設定への影響も記述すべき。

#### 1.2.9 「全データアクセスScalarDB経由」制約の強調が不十分

**修正提案**: この制約はデシジョンツリーにも明記すべき。「ScalarDB導入時の前提条件」として独立セクションを設け、3.17のTransaction Metadata Decouplingによる部分的緩和も併記すべき。

#### 1.2.10 セカンダリインデックスの結果整合性への再定義の影響

**修正提案**: Value Propositionの「ACID保証」主張に但し書きを追加し、ACID保証の正確なスコープ（プライマリキーアクセスベース）を明記すべき。

### 1.3 重大な誤りや見落とし

#### TCCパターンの提唱者の誤記

**箇所**: 07_transaction_model.md セクション5.1「Pat Helandが提唱」

**指摘**: Pat Helland（スペルミス）が提唱したのは「Life beyond Distributed Transactions」であり、TCCパターンそのものではない。AtomikosのGuy Pardonらによる体系化が知られている。

#### ScalarDBの書き込みレイテンシの定量データ欠如

Consensus Commitプロトコルの絶対的なパフォーマンスオーバーヘッドの数値（「ScalarDBなし vs あり」）が3.17ベンチマーク以外に提示されていない。

#### マルチリージョン構成の現実的な制約

Coordinatorテーブルのリージョン間レイテンシ影響の具体的な分析がない。100ms〜数百msオーダーの変動がユースケース判定を修正する可能性がある。

### 1.4 3.17新機能がValue Propositionに与える影響

| 機能 | Value Propositionへの影響 | 影響度 |
|------|--------------------------|--------|
| Piggyback Begin + Write Buffering | パフォーマンスオーバーヘッドの弱点を大幅に緩和。Indirect modeで約2倍改善 | 高 |
| Batch Operations | API利便性とパフォーマンスの両面で強化 | 中 |
| Transaction Metadata Decoupling | **Value Propositionを根本的に拡張する最重要機能**。既存DBのスキーマ変更不要でDB移行ユースケースの実現可能性が飛躍的に向上 | 極めて高 |
| Multiple Named Embedding Stores | AI/LLM時代のユースケースを開拓 | 中（将来的に高） |
| Secondary Index Fix | ACID保証のスコープ明確化（結果整合性の導入）。マーケティングメッセージに注意が必要 | 中（ネガティブ方向にも注意） |

## 2. マイクロサービスアーキテクチャ レビュー

**総合評価: 4 / 5（良好）**

**対象ファイル**: 01_microservice_architecture.md, 03_logical_data_model.md, 07_transaction_model.md, 08_transparent_data_access.md

### 2.1 良い点

- **DDDの境界コンテキストを最重要指針と位置づけ**: サービス境界設計の第一原則として正統的な設計方針
- **データ所有権の原則が明確**: 「各データエンティティは唯一のサービスがオーナー（Single Source of Truth）」と明記。ECサイトの具体例でオーナーシップマッピングまで提示
- **データ共有パターンの選択肢が豊富**: API呼び出し、イベント駆動ローカルコピー、参照IDのみ保持など結合度の異なる選択肢を提示
- **Saga vs 2PCの選択基準が明確**: 「強整合性が必要 → ScalarDB 2PC」「結果整合性で可 → Saga」という正しい設計指針
- **TCCパターンの詳細な解説**: Saga、2PC、TCCの3パターンを比較し、ScalarDB 2PCがTCCのTry/Confirm/Cancelフローに本質的に対応することを指摘
- **Outboxパターンの二重書き込み問題の解説が的確**: ScalarDBの異種DB間での原子性保証の位置づけが正確
- **API選定ガイドが優秀**: REST（外部公開）、gRPC（内部通信）、GraphQL（BFF）、Kafka/RabbitMQ（非同期）の使い分けが適切
- **障害復旧パターンが体系的**: サーキットブレーカー、リトライ（指数バックオフ）、タイムアウト、バルクヘッド、フォールバック、冪等性の6パターンをカバー

### 2.2 改善が必要な点

#### 2.2.1 ScalarDB Shared Clusterにおけるデータ所有権の曖昧化リスク

**問題**: Shared-Clusterパターンを「推奨」としているが、複数のマイクロサービスが同一のScalarDBインスタンスを共有するため、実質的にShared Databaseパターンへの回帰リスクがある。

**修正提案**: Shared-Clusterパターンの採用条件を明示すべき:
1. NamespaceレベルでのRBAC（3.17）を必ず適用し、サービス間のアクセスを制限
2. ScalarDB Cluster自体のバージョンアップ時に全サービスに影響が及ぶため、Blue-Greenデプロイ戦略を必須とする
3. ScalarDB Clusterの障害が全サービスの障害に直結するため、HA構成（最低3ノード）を必須とする
4. 「データベースインフラの共有」であり「データの共有」ではないことを設計原則として明示

#### 2.2.2 Separated-Clusterパターンにおけるサービス間結合の評価不足

**問題**: 2PCは本質的に同期的であり、全参加者の可用性を要求する。これは「自律性」原則と矛盾する。

**修正提案**: 以下の記述を追加すべき:
- 2PCに参加する全サービスが同時に稼働している必要がある
- 2PCの使用範囲を「本当に強整合性が必要な箇所」のみに最小限化
- 2PC不要な操作はイベント駆動で非同期化
- 2PCのタイムアウトとフォールバック戦略を必ず設計

#### 2.2.3 ScalarDB Clusterの単一障害点リスク分析の欠如

**修正提案**: 以下を追加すべき:
- Shared-Cluster障害 → 全サービスのトランザクション処理停止の影響範囲分析
- Circuit Breakerによるグレースフルデグラデーション設計
- 読み取り専用のフォールバックモード設計
- ネットワーク分断時のOCC挙動（トランザクションabort）の説明

#### 2.2.4 コンテキストマッピングの具体的な定義の欠如

**修正提案**: ScalarDB 2PC使用時のコンテキスト間関係を定義すべき:
- Coordinatorサービス = Upstream
- Participantサービス = Downstream
- Anti-Corruption Layer: Participant側で2PC固有プロトコルがドメインロジックに浸透しないよう設計
- Open Host Service: CoordinatorがトランザクションIDをPublished Languageとして公開

#### 2.2.5 CDC/Debeziumの具体的なフィルタリング設定例の不足

**修正提案**: メタデータカラム（`tx_id`, `tx_state`, `tx_version`等）のフィルタリング方法について、Debeziumのトランスフォーメーション設定例を追加すべき。

#### 2.2.6 非機能要件に対するScalarDBオーバーヘッドの定量化不足

**修正提案**: 以下の目安を追加すべき:
- 単一レコード操作: ネイティブDBに対して約1.5-3倍のレイテンシ
- マルチストレージトランザクション: 参加ストレージ数に比例して増加
- 2PCトランザクション: 参加サービス数 × ネットワークRTTが追加
- 超低遅延要件（< 10ms）のケースではキャッシュやマテリアライズドビューからの読み取りを推奨

### 2.3 重大な設計上の問題

#### 問題1: 分散モノリスへの傾倒リスク（深刻度: 高）

ScalarDB 2PCの利便性を強調するあまり、MSAの本質的な利点（独立性、障害隔離、チーム自律性）が2PCの同期的結合によって損なわれるリスクへの言及が薄い。

**具体的なリスク**:
1. ランタイム結合: 全参加サービスが同時に稼働していなければトランザクション失敗
2. デプロイメント結合: ScalarDB ClusterやAPIの変更が全参加サービスに影響
3. 障害の伝播: 1サービスの遅延が2PC全体を遅延させる
4. チーム間の協調コスト増加

**対策提案**: 「2PC適用の制限ガイドライン」を明記すべき:
- デフォルトは結果整合性（Saga/イベント駆動）とする
- 2PCの適用許可条件: ビジネス上一時的不整合が法規制・金銭的損失に直結、参加サービスが2つ以下、同一チーム内管理、実行時間100ms以下
- 2PC使用時の必須対策: タイムアウト設定、Circuit Breaker、フォールバック戦略、監視ダッシュボード

#### 問題2: ScalarDB経由の排他的データアクセス制約の影響分析不足（深刻度: 高）

**具体的なリスク**:
1. 技術的多様性の制限: ORM（JPA/Hibernate等）やDB固有機能の利用制限
2. ベンダーロックイン: ScalarDBからの移行コストが極めて高い
3. 運用上の制約: DBA直接SQLによる緊急データ修正がScalarDBの整合性を壊す可能性

**対策提案**:
- ScalarDB管理対象の最小化原則（サービス間トランザクションに参加するテーブルのみ）
- 緊急時データ修正手順の事前整備
- 脱退戦略（Repository層によるScalarDB抽象化）

#### 問題3: Shared-ClusterとDatabase per Service原則の矛盾（深刻度: 中）

**具体的な問題**:
- Coordinatorテーブルが全サービス共有のボトルネック化リスク
- ノード障害・OOMの全サービスへの波及
- Noisy Neighbor問題

**対策提案**:
- ResourceQuotaで十分なリソース確保
- Group Commit有効化でCoordinatorテーブル最適化
- Shared → Separated Clusterへの段階的移行計画を事前策定

#### 6.2.1 Consensus Commitレイテンシの内訳分解が不在

**追加すべき内容**: 各フェーズ（Begin/Read/Write/Prepare/Validate/Commit）のレイテンシ内訳と合計見積もり
- Snapshot Isolation（最適化なし）: ~20-53ms
- Snapshot Isolation（全最適化あり）: ~14-35ms

#### 6.2.2 OCC競合率のモデリングと閾値が欠落

**追加すべき内容**:
- 競合確率の近似式: `P ≈ 1 - (1 - k/N)^(C-1)`
- 実用的閾値: P < 5%（良好）、5-15%（注意）、15-30%（危険）、> 30%（設計変更必須）
- 設計時チェックリスト: アクセス集中度定量化、ピーク同時Tx見積もり、事前競合率計算

#### 6.2.3 Group Commitの詳細パラメータとチューニング指針

**追加すべき内容**:
- `slot_capacity`（デフォルト20、高スループットでは40-80）
- `delay_duration_millis`（低レイテンシなら2-5ms、高スループットなら10-20ms）
- スループット改善 vs レイテンシ増加のトレードオフ曲線

#### 6.2.4 ネットワークレイテンシの多段ホップ分析

**追加すべき内容**:
- 同一AZ: App → Envoy → ScalarDB → DB = 合計RTT 3-8ms
- マルチAZ: 4-12ms
- マルチリージョン: 42-206ms
- gRPC接続プーリング設計指針（min_idle/max_idle/max_total）

#### 6.2.5 水平スケーリングの限界とPod数指針

**追加すべき内容**:
- スケーリング限界要因: Coordinatorテーブルのボトルネック、基盤DBコネクション上限、OCC競合率増大
- Pod数設計: `Pod数 × parallel_executor_count ≤ DB max_connections × 0.8`
- HPA設定推奨: CPU使用率70%、maxReplicasはDB接続上限から逆算

**追加すべき内容**:

**追加すべき内容**:
- 固定メタデータ: ~80-100 bytes/record
- before-image: アプリデータサイズ × 1.0

| Coordinatorテーブルがスケーリングボトルネック | 高 | 書き込みスループット上限計測、TTL設定設計 |
| OCC競合によるリトライストーム | 高 | 競合率モニタリング、閾値アラート（10%警告、20%停止） |
| クロスパーティションスキャンの暗黙的性能劣化 | 中〜高 | 検出メトリクス定義、設計レビューチェックリスト |
| Metadata Decoupling（Virtual Table）の読み取り性能 | 中 | 有効/無効でのRead/Write性能比較ベンチマーク |
| バッチ処理のTxタイムアウトとチャンクサイズ不整合 | 中 | タイムアウトデフォルト値明記、チャンクサイズとの関係定式化 |

### 6.4 ベンチマーク推奨事項

**SLI/SLO推奨**:
- 単一レコードGet: p99 < 50ms
- 単一レコードWrite（Commit含む）: p99 < 200ms
- トランザクション成功率: > 99.5%
- OCC競合率: < 5%（正常）、< 15%（許容上限）

**負荷テスト変数**:
1. 同時スレッド数: 1, 16, 64, 128, 256, 512
2. Read/Write比率: 9:1, 5:5, 1:9
3. トランザクションサイズ: 1, 5, 20 操作/Tx
4. パーティション競合度: 低(UUID PK), 中(1000), 高(10)
5. ネットワーク構成: Direct mode, Indirect mode, マルチAZ

### 6.5 その他の補足指摘

- `scan_fetch_size`のチューニング指針が不足（バッチ処理では100-1000への増加が有効だがRead Set増大リスクとのバランスが必要）
- `async_commit`の耐久性リスク（コミット失敗の可能性）と許容ユースケースの境界が未明記
- One-Phase Commitの有効条件（単一パーティション操作のみ）とリスクが未説明
- JVM GC（G1GC/ZGC選択）への影響が未記載（大きなRead/Write SetによるGC pauseリスク）

## 7. 可用性 レビュー

**総合評価: 4 / 5（良好）**

**対象ファイル**: 12_disaster_recovery.md, 06_infrastructure_prerequisites.md, 11_observability.md

### 7.1 良い点

- **Consensus Commitの障害回復が詳細**: Coordinatorテーブルの役割、Lazy Recoveryの動作原理、各クラッシュケースの回復フローが明確
- **バックエンドDB別のバックアップ戦略が具体的**: DynamoDB/Cosmos DB/Cassandra/RDS・Aurora別の一時停止要否、PITR対応、リストア手順が整理。特にCosmos DBリストア後の`--repair-all`注意点は実運用で見落としやすいポイント
- **Kubernetes HA構成が実践的**: PDB/Anti-Affinity/Topology Spread Constraints/Taint・Tolerationの設定例がYAMLで提示
- **監視スタックが体系的**: Prometheus + Grafana + Loki、組み込みPrometheusRuleアラート、カスタムアラートルール、エスカレーションポリシーまで一貫
- **RPO/RTOが障害シナリオ別に整理**: Pod障害からリージョン障害まで表形式

### 7.2 考慮不足な点

#### 7.2.1 ScalarDB Cluster可用性

**Readiness Probe未定義のリスク**: 起動直後でバックエンドDB接続未完了のPodにトラフィック転送 → 大量Tx失敗リスク
- 推奨設定（gRPC Health Check + DB接続確認）の追加
- `minReadySeconds`を10-30秒に設定

**コンシステントハッシュリング再計算中のリクエスト影響**: インフライトリクエストの扱い、クライアントSDKのキャッシュ遅延、direct-kubernetes vs indirectモードの差異が不明確

#### 7.2.2 Coordinatorテーブル

**単一障害点リスクの定量的評価が不足**:
- 利用不能時の全書き込みTx停止、Lazy Recovery停止の影響範囲
- Coordinatorテーブル用DBの推奨構成（レプリケーション、フェイルオーバー設定）
- 古いエントリのクリーンアップ戦略（TTL、定期パージ）とサイズ監視

#### 7.2.3 基盤DB可用性との整合

**マネージドDBのメンテナンスウィンドウ影響**:
- RDS/Auroraメンテナンス中のTx失敗パターンと対策
- DynamoDBスケーリング中のスロットリング影響
- Cosmos DB RU枯渇時の429エラーハンドリング

**DBフェイルオーバー時のコネクション再接続**:
- JDBCコネクションプール全接続のフェイルオーバー前エンドポイント保持問題
- DNSキャッシュTTLとフェイルオーバー検知遅延
- Auroraフェイルオーバー中のエラー数見積もり

#### 7.2.4 Lazy Recovery

**アクセスされないレコードのPREPARED状態残存問題**:
- 後続Txでのロールフォワード/ロールバック実行によるレイテンシ増大
- 定期クリーンアップジョブの設計（能動的回復）
- 高トラフィック時のLazy Recovery連鎖的遅延

**トランザクション有効期限（15秒）の調整ガイダンス**: 長時間Tx延長のリスクとトレードオフ

#### 7.2.5 マルチAZ/リージョン

**フェイルオーバー判定基準の曖昧さ**:
- 自動/手動フェイルオーバーの判断フローチャート
- フェイルオーバー中の未レプリケーションTxデータ損失量の計測方法
- スプリットブレイン防止策

**Remote Replicationのレプリケーションラグ監視**: メトリクス、アラート、ラグベースのRPO動的計算

#### 7.2.6 バックアップ・リストア

**Multi-storage構成のバックアップ整合性**:
- 異なるDB（DynamoDB + RDS等）のPITRリストアポイント同期手順
- クロック精度差異によるリストアポイントずれの見積もり

**一時停止中のサービス影響**: クライアントリクエストの振る舞い、タイミング最適化、一時停止失敗時リカバリ

**リストア後の検証手順**: Coordinator↔アプリテーブル整合性検証スクリプト、Lazy Recovery完了確認

#### 7.2.7 未記載の障害パターン

| 障害パターン | 追加すべき内容 |
|-------------|--------------|
| ライセンスキー期限切れ | 動作（即停止/猶予期間）、監視方法、更新手順 |
| TLS証明書期限切れ | gRPC接続不可、cert-manager自動更新監視 |
| K8sコントロールプレーン障害 | メンバーシップ凍結の影響 |
| Envoyプロキシ障害（indirect） | トラフィック遮断と復旧、Envoy自体のHA |
| ディスク枯渇 | バックエンドDBディスクフル時のTx失敗パターン |
| OOMKill | 4GBメモリ上限でのOOMKill影響、ヒープ推奨値 |
| 設定ミスデプロイ | ダウングレード不可制約下のロールバック手順 |
| Coordinatorテーブル破損 | 全Tx停止リスクと復旧手順 |

**アップグレード中の障害**: 新旧バージョン混在時の互換性、中断・ロールバック手順、カナリアデプロイ戦略

### 7.3 重大な可用性リスク

| リスク | 重大度 | 影響 |
|--------|--------|------|
| Coordinatorテーブルの可用性がシステム全体を律速 | 高 | 利用不能で全書き込みTx停止 |
| Readiness Probe未定義による起動時Tx損失 | 中〜高 | ローリングアップデート/スケールアウト時のTx大量失敗 |
| ダウングレード不可による復旧選択肢制限 | 中 | RTO大幅増大 |
| K8sコントロールプレーン障害時のメンバーシップ管理停止 | 中 | 新規ノード追加・障害ノード検知不可 |
| 非同期レプリケーション下のフェイルオーバー時データ損失 | 中 | レプリケーションラグ分のTx喪失 |

### 7.4 DR演習への推奨事項

#### 段階的DR演習プログラム

| フェーズ | 内容 | 所要期間 |
|---------|------|---------|
| Phase 1 | コンポーネント単体試験 | 1-2週間 |
| Phase 2 | 統合障害試験（複合障害シナリオ） | 1-2週間 |
| Phase 3 | バックアップ・リストア完全試験 | 1週間 |
| Phase 4 | リージョンフェイルオーバー試験 | 1週間 |
| Phase 5 | 定期的Chaos試験（自動化） | 継続 |

#### ScalarDB固有の検証シナリオ

- CoordinatorテーブルDB障害 → 全Tx回復時間計測
- 大量PREPAREDレコード残存からのLazy Recovery性能検証
- 一時停止バックアップの完全実行と一時停止時間・RTO計測
- ローリングアップデート中のTx成功率・レイテンシ変化計測

#### SLA設計の補足

- 99.95% HAパターンではRDS Multi-AZフェイルオーバー（数分）で月間許容ダウンタイム21.9分の大半を消費するリスク
- 99.99% DRパターンでは非同期Remote Replication + Active-Active未サポートにより月間4.3分の許容ダウンタイムは達成困難
- バックエンドDB SLA（Aurora 99.99%等）とScalarDB Clusterの複合SLA算出方法の追加が必要
